# 02 - 开发方法论 (Methodology)

## 推荐：迭代式开发 (Iterative Development)
## 辅以精益 "构建-测量-学习" 循环 + 原型快速验证精神

---

## 为什么不是其他方式？

| 方式 | 评价 | 为什么不选 |
|------|------|-----------|
| **瀑布 (Waterfall)** | ❌ | 要求前期定死需求。独立游戏开发视觉/交互经常调，瀑布会让你半年后才发现方向错了。 |
| **螺旋 (Spiral)** | ❌ | 风险驱动 + 每个循环正式风险评估。一个人做这些仪式是浪费时间。 |
| **原型 (Prototype)** | ❌ | Throwaway 原型写完就扔。对独立开发者，每一行能用的代码都是资产，不能丢。 |
| **RAD** | ❌ | 快速应用开发，偏企业信息系统，依赖 CASE 工具和用户高频协同评审。 |
| **JAD (联合应用开发)** | ❌ | 需要团队 workshop 和决策者在场。一个人不存在"联合"。 |
| **敏捷 (Agile)** | ⚠️ | 太泛，是价值观不是具体流程。Scrum 和 XP 都属于敏捷，我们取其精神但不用其重量。 |
| **XP (极限编程)** | ❌ | 结对编程（一个人没法做）、TDD（Lua 生态测试工具不成熟，新人学 Lua 同时学 TDD 负担太重）、持续集成（项目初期没必要）。 |
| **Scrum** | ❌ | Sprint planning、每日站会、回顾会、燃尽图。一个人开这些会纯属内耗。 |
| **精益 (Lean)** | ⚠️ | "消除浪费"和"构建-测量-学习"非常好，但它是哲学不是具体流程。我们把精益的思想融入迭代。 |
| **DevOps** | ⚠️ | 是运维/发布文化不是开发方式。V5 做发布自动化时自然会用到。 |
| **增量 (Incremental)** | ⚠️ | 和迭代高度互补，但增量强调"分块交付功能"，迭代强调"逐步精化产品"。我们的 V0-V5 本身就是分块增量交付，但核心精神是迭代精化。 |

## 为什么选迭代？

1. **每轮都有可运行产物**：V0 到 V5，每个版本结束时 `love .` 都能跑起来，都有明确视觉成果。不会写三个月代码还见不到东西。
2. **打 tag 自由回滚**：每个版本一个 git tag (`v0.0.0`, `v1.0.0`...)，随时切换验证。这天然支持独立开发者"我觉得 V3 搞砸了，回到 V2 接着做"。
3. **架构允许自然演进**：不一开始做"大架构"。Core 层在 V0 锁定接口，Engine 层在需要时添加，Game 层在 V1+ 才开始。每个迭代只写当前需要的，不过度设计。
4. **轻量无仪式**：没有会议，没有故事点，没有燃尽图。只有：写 → 跑 → 验 → 提 → tag → 下一轮。
5. **精益反馈循环**：每个版本完成后你自己玩 5 分钟，觉得哪里别扭，下一个版本修。这就是"构建-测量-学习"。
6. **完美适配 solo 节奏**：你一个人可以决定"V2 多花两天做动画"或者"AI 太难了先做简单版"。不需要跟任何人汇报或协商。

## 迭代节奏（实际操作流程）

```
每个版本 Vn：
  1. 阅读 docs/04-roadmap.md 中 Vn 的目标/范围/验收标准
  2. 从 main 创建分支或直接在 main 开发
  3. 写代码 → 跑 love . → 修 bug → 跑 love .
  4. 对照验收标准逐项检查
  5. 写 CHANGELOG（这个版本做了什么）
  6. git commit
  7. git tag -a vN.0.0 -m "VN: ..."
  8. git push && git push --tags
  9. 休息，玩自己的游戏，记下下版本要改的地方
```

## 版本号规则

采用语义化版本 (SemVer)：`vMAJOR.MINOR.PATCH`
- MAJOR：架构/API 不兼容变更
- MINOR：新增功能，向后兼容
- PATCH：bug 修复

V0-V5 每个大版本发布时 MINOR=0, PATCH=0，即 v0.0.0, v1.0.0, v2.0.0...
中途小改打 v0.1.0, v0.0.1 等。
